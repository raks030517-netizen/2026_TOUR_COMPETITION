# 부산 AI 대화형 여행 서비스 잔여 기능 단계별 구현 프롬프트

현재 저장소의 부산 AI 대화형 여행 서비스 프로젝트를 계속 개발해 주세요.

먼저 아래 현재 구현 상태를 기준선으로 사용하세요.

## 현재 구현 상태

### 완료된 기능

* 회원가입
* 로그인
* 로그아웃
* 회원 중복 확인
* 로그인 세션 유지
* BCrypt 비밀번호 암호화
* 인증 API 보호
* 네이버 Dynamic Map SDK 로딩
* 일반 장소 마커 표시
* 현재 위치 마커 표시
* 경로 Polyline 표시
* 현재 위치 기반 공공데이터 날씨 API
* 네이버 Directions 자동차 길찾기
* ODsay 대중교통 길찾기
* ODsay 도보 길찾기
* 경로 상세 정보 표시
* 네이버 지역 검색 API
* 공공 관광 API
* 관광지·맛집 검색 결과 지도 표시
* 현재 화면의 위치·장소·경로 정보를 참고한 기초 AI 채팅

### 부분 구현된 기능

* 현재 위치는 `getCurrentPosition`으로 한 번 조회할 수 있지만 `watchPosition` 기반 연속 추적은 없음
* 장소 추천에 위치 일부는 반영되지만 날씨와 여행 일정은 추천 문맥에 완전히 연결되지 않음
* AI는 현재 화면 정보를 프롬프트로 받아 답변하지만 실제 Tool Calling은 없음

### 미구현 기능

* 여행 일정 엔티티와 테이블
* 여행 일정 CRUD API
* 여행 일정 화면
* 사용자별 여행 데이터 저장
* 사용자별 채팅 저장
* AI Tool Calling
* AI 응답에 따른 지도 직접 제어
* 위치·시간·날씨·이동시간 기반 일정 분석
* AI 일정 변경안
* 사용자 승인 후 일정 변경
* 여행 기록 자동 생성
* 여행 기록 저장 및 조회

현재 DB에는 `app_users` 테이블만 존재합니다.

완료된 기능을 처음부터 다시 구현하지 마세요.

기존 정상 기능을 유지하면서 필요한 부분만 확장하세요.

# 1. 최종 목표

최종적으로 다음 사용자 흐름이 실제로 동작해야 합니다.

```text
회원가입 및 로그인
→ 여행 생성
→ 여행 날짜별 일정 작성
→ 관광지·맛집을 검색하여 일정에 추가
→ 현재 위치 확인
→ 현재 날씨 확인
→ 자동차·대중교통·도보 경로 조회
→ AI에게 주변 장소 추천 요청
→ AI가 실제 Tool을 통해 장소·날씨·일정·경로 조회
→ AI 추천 결과가 지도 마커에 반영
→ AI가 현재 일정의 문제점을 분석
→ AI가 일정 변경안을 제안
→ 사용자가 변경안을 확인하고 적용
→ 방문 장소 완료 처리
→ AI가 실제 방문 정보를 기반으로 여행 기록 초안 생성
→ 사용자가 수정하고 저장
→ 로그아웃 후 다시 로그인해도 여행·일정·대화·기록 유지
```

# 단계 1: watchPosition 기반 실시간 위치 추적 완성

사용자가 `다음 단계 진행` 또는 `1단계 진행`이라고 입력했을 때 수행하세요.

## 목표

현재 `getCurrentPosition`으로 한 번만 조회하는 위치 기능을 `watchPosition` 기반 연속 위치 추적으로 확장합니다.

## 필수 기능

* 초기 위치 조회
* `watchPosition` 시작
* 위치 변경 시 기존 현재 위치 마커 갱신
* 새 위치가 들어올 때마다 새 마커 생성 금지
* 위치 정확도 관리
* 위치 조회 시각 관리
* 위치 권한 거부 처리
* 위치 기능 미지원 처리
* 위치 조회 시간 초과 처리
* 일시적 위치 오류 처리
* 마지막 정상 위치 유지
* 페이지 이탈 시 `clearWatch`
* 컴포넌트 재렌더링 시 중복 watch 방지
* 로그인 화면 등 위치가 필요 없는 화면에서는 watch 미실행

## 위치 상태 구조

```json
{
  "latitude": 35.1796,
  "longitude": 129.0756,
  "accuracy": 20,
  "observedAt": "2026-07-16T11:00:00+09:00"
}
```

위치 미확인 상태에서 `0, 0` 좌표를 사용하지 마세요.

# 단계 4: AI Tool Calling 기반 구조

사용자가 `다음 단계 진행` 또는 `4단계 진행`이라고 입력했을 때 수행하세요.

## 목표

현재 화면 정보를 프롬프트 문자열에 넣는 방식에서 벗어나, AI가 필요한 실제 데이터를 Tool을 통해 조회하도록 만듭니다.

Spring AI가 이미 사용 중이라면 현재 버전에 맞는 Tool Calling 방식으로 구현하세요.

현재 라이브러리 버전의 공식 API를 확인하지 않고 이전 버전 문법을 추측하지 마세요.

## 필요한 Tool

```text
getCurrentLocation
getCurrentWeather
getTrip

searchPlaces
getRoute
```

## Tool 역할

### getCurrentLocation

현재 프런트엔드가 전달한 최신 위치 문맥을 조회합니다.

브라우저 위치를 백엔드가 자동으로 알 수 없으므로, 채팅 요청 DTO에 현재 위치를 명시적으로 포함하거나 안전한 현재 문맥 저장 방식을 사용하세요.

### getCurrentWeather

기존 날씨 Service를 호출합니다.

AI가 날씨 API를 직접 호출하지 않게 하세요.

### getTrip

현재 로그인 사용자의 여행 정보를 조회합니다.

사용자 소유권 검증이 반드시 적용되어야 합니다.

### getTodayItinerary

해당 여행과 날짜의 일정, 방문 상태, 예정 시간을 조회합니다.

### searchPlaces

기존 네이버 지역 검색·관광 API Service를 재사용합니다.

장소명, 주소, 좌표를 AI가 생성하지 않게 하세요.

### getRoute

기존 자동차·대중교통·도보 Route Service를 재사용합니다.

## Tool 호출 보안

* AI가 임의의 userId를 지정하지 못하게 하기
* 로그인 사용자는 인증 Context에서 결정
* tripId 소유권 검증
* 위도·경도 검증
* 외부 API Key AI에 전달 금지
* 엔티티 원본 전체를 Tool 응답으로 노출하지 않기
* 필요한 DTO만 반환

# 단계 8: 사용자 승인 후 일정 변경

